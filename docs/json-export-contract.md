# JSON Export Contract

`ontology.json` is generated by `scripts/build_json.py` from the newest
`Ontology_v*.csv` file in the repository root. It is intended for tools that
should not parse the quoted, multi-line CSV directly.

## Top-Level Shape

```json
{
  "description": "Machine-readable export of the Active Inference Ontology.",
  "source": "Ontology_v5_May_25_2023.csv",
  "termCount": 429,
  "tagCount": 8,
  "edgeCount": 238,
  "tags": [],
  "terms": [],
  "graph": {
    "nodes": [],
    "edges": []
  }
}
```

Counts are derived at build time. Do not edit them manually.

## Term Records

Each `terms[]` entry has this shape:

| Field | Meaning |
| --- | --- |
| `id` | Stable derived id as long as the term label is unchanged. |
| `term` | Human-readable term label from the CSV. |
| `tag` | Conceptual area from the CSV `Tag` column. |
| `list` | Source list grouping from the CSV `List` column. |
| `definition` | Normalized `Proposed Definition 1`. |
| `definition2` | Normalized `Proposed Definition 2`. |
| `examples` | Normalized `Correct Examples`. |
| `counterExamples` | Normalized `Incorrect Examples`. |
| `connections` | Normalized raw connection prose. |

The builder normalizes repeated whitespace in all string fields. It does not
perform semantic validation of definitions or examples.

## Graph Nodes

Each `graph.nodes[]` entry is derived from a term:

| Field | Meaning |
| --- | --- |
| `id` | Same id as the source term record. |
| `label` | Term label for display. |
| `type` | Tag name, or `Other` when the tag is empty. |
| `meta` | First 160 characters of the primary definition, or the tag if the definition is empty. |

Graph nodes are suitable for dashboards, concept maps, and search indexes.

## Graph Edges

Each `graph.edges[]` entry is derived from a detected term mention in the source
term's `Connections` field:

| Field | Meaning |
| --- | --- |
| `source` | Source term id. |
| `target` | Referenced term id. |
| `relation` | Currently always `connects`. |

Edges are intentionally minimal. Relation typing is a future enhancement because
the current source connection field is free text.

## Compatibility Notes

- Consumers should treat unknown top-level fields as additive.
- Consumers should not rely on array order except for deterministic display.
- Consumers should use `id` for joins and `term` for display.
- Consumers should expect future releases to add fields such as aliases,
  relation types, review metadata, or provenance.
